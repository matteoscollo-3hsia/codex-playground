## Comando core della pipeline

```bash
uv run python run.py create-input --from-hubspot "ACME"
```

Questo e' il vero entrypoint operativo della pipe: recupera il lead input da HubSpot via Private App e avvia il flusso HubSpot -> LeadInput -> Excel -> OpenAI -> DOCX.

```bash
uv run python run.py generate-primer --lead-input "primers/NOME CLIENTE/_dossier/lead_input.json"
```

Questo e' il comando operativo successivo piu generico: permette di indicare esplicitamente il path del `lead_input.json` da usare per la generazione del primer, cambiando solo il nome del cliente nella cartella.

## Aggiornamento post-review

Le sezioni analitiche sotto riflettono il review snapshot originario; questo blocco annota gli aggiornamenti successivi gia' implementati.

- `lead_input.json` include ora il campo `deal_notes`.
- `create-input --from-hubspot` recupera le vere HubSpot Notes associate al deal selezionato, leggendo `hs_note_body` dagli oggetti `notes`, non la `deal description`.
- La selezione del deal privilegia l'intersezione company + contatto primario, poi altri contatti associati, poi fallback su un deal della company.
- Il completamento interattivo dei campi mancanti include anche `deal_notes`.
- I `lead_input.json` vengono salvati in `OUTPUT_BASE_DIR/<company_folder>/_dossier/lead_input.json`; in questa repo il template pratico e' `primers/NOME CLIENTE/_dossier/lead_input.json`.
- Per una sezione come `1.2 What we know so far`, il modo corretto e' usare `{{deal_notes}}` come contesto esclusivo e vietare esplicitamente il mixing con la web research.
- Sono stati aggiunti test unitari dedicati per il mapping HubSpot deal -> notes -> `LeadInput`.

## 1. Executive Summary

Il progetto è **ben strutturato e funzionalmente solido** per la fase implementata. L'architettura modulare (src layout, helper separati, Pydantic per i modelli) è appropriata per un tool CLI di questa complessità. La pipeline core — HubSpot → LeadInput → Excel → OpenAI → DOCX — è implementata e funzionante.

### Punti di forza
- ✅ Architettura modulare chiara e coerente con il design dichiarato in `CLAUDE.md`
- ✅ Type hints moderni (Python 3.10+ union syntax), nessun import inutilizzato, zero import circolari non gestiti
- ✅ Scritture atomiche (`.tmp` → `rename`) + multi-path output (`latest/` + `runs/`)

### Aree critiche
- 🔴 **Silent I/O failures**: `io_helpers.py` inghiotte `OSError` silenziosamente — un primer può non essere scritto senza che l'utente lo sappia

- 🔴 **Test coverage insufficiente**: 4 aree core a 0% di copertura (retry logic, HubSpot, sheet filtering, citation extraction)

### Maturità stimata: **62%**
La fase 1 è funzionale e deployabile per uso interno con le limitazioni note. Non è ancora production-ready per un uso non supervisionato.

### `excel_helpers.py` — ⚠️ Performance e coerenza

**Scopo:** Anchor lookup su Excel, sostituzione placeholder (4 sintassi), parsing titoli step.

**Problemi:**

| Issue | Severità | Descrizione |
|-------|----------|-------------|
| `_find_anchor_exact` O(n×m) | 🟢 Basso | Scansione lineare di tutte le celle per ogni chiamata. Su file grandi (50+ fogli, centinaia di righe) può diventare lento. Soluzione: cache dict `{normalize(cell.value): cell}` costruita una volta per foglio |
| `re.escape` inconsistente | 🟡 Medio | Solo la sintassi `<< k >>` usa `re.escape(str(key))` per il valore del placeholder; le altre 3 sintassi usano f-string dirette. Se una chiave contiene regex speciali (`.`, `*`, `+`), le prime 3 sintassi possono fare match sbagliati |
| `_first_right_value(limit=6)` | 🟡 Medio | Magic number 6 — non documentato. Perché 6 colonne? |

**Nota positiva:** l'ordinamento per lunghezza chiave (`sorted(..., key=lambda k: len(str(k)), reverse=True)`) evita correttamente la sostituzione parziale (es. `name` dentro `company_name`). Logica smart ma non documentata.

### `openai_helpers.py` — ⚠️ Magic values e duplicazioni

**Scopo:** Chiamate API OpenAI, retry con exponential backoff, estrazione citazioni.

**Problemi:**

| Issue | Severità | Descrizione |
|-------|----------|-------------|
| `_REQUEST_TIMEOUT_SECONDS = 30 * 60` | 🟠 Alto | 30 minuti hardcoded, non configurabile via env var |
| `_model_supports_reasoning_effort()` | 🟡 Medio | Controlla solo `model.startswith("gpt-5")` — brittle se nuovi modelli con reasoning hanno altri prefissi |
| `_is_model_not_found_error()` | 🟡 Medio | Logica `"model" in msg` ripetuta 3 volte per `isinstance(NotFoundError)`, `status_code == 404`, e fallback stringa — consolidabile in 3 righe |
| Retry cap 10s hardcoded | 🟢 Basso | `min(base * 2^attempt, 10.0)` — il cap a 10s non è configurabile |
| URL regex fragile | 🟢 Basso | `https?://[^\s)>"]+` — non cattura URL con parentesi bilanciate o query string complesse |

**Nota positiva:** il pattern `_extract_output_text_from_response` con duck typing (dict vs object) è robusto per variazioni di formato risposta OpenAI.

### `io_helpers.py` — 🔴 CRITICO

**Scopo:** Scritture atomiche su disco (`.tmp` → `rename`).

**Problema critico:**

```python
# riga ~13
except OSError:
    return  # Nessun logging — failure totalmente silenzioso
```

Se il disco è pieno, se i permessi mancano, o se il rename fallisce, il primer non viene scritto **ma l'esecuzione termina con "successo"**. L'utente non riceve nessuna indicazione. Lo stesso vale per `_safe_write_json` che inghiotte `TypeError` e `ValueError`.

**Fix immediato richiesto:**
```python
import logging
logger = logging.getLogger(__name__)

except OSError as exc:
    logger.warning("Failed to write %s: %s", path, exc)
    return
```

Questo è già documentato in `CLAUDE.md` come "Known Issue #3" ma non ancora risolto.

### `hubspot_client.py` — ⚠️ Error handling asimmetrico

**Scopo:** Ricerca aziende HubSpot, fetch contatti, selezione deal associato, fetch delle vere HubSpot Notes del deal, mapping finale a `LeadInput`.

**Aggiornamento implementato:** il client non usa la `deal description`; legge gli oggetti `notes` associati al deal e concatena `hs_note_body` in `deal_notes`.

**Problemi:**

| Issue | Severità | Descrizione |
|-------|----------|-------------|
| No token validation upfront | 🟠 Alto | `HUBSPOT_TOKEN` non viene validato prima della prima chiamata — un token non configurato produce un 401 HTTP a runtime, senza messaggio chiaro all'utente |
| Error handling asimmetrico | 🟠 Alto | `get_associated_contacts()` gestisce silenziosamente il 404 (ok), ma `search_companies()` non ha gestione 404 — comportamenti diversi per errori simili |
| Nessun retry logic | 🟠 Alto | Rate limit HubSpot (100 richieste/10s per app) non gestito — su volumi alti produrrebbe eccezione non handled |
| Infinite loop menu selezione | 🟡 Medio | `_pick_company()` usa `while True` senza escape — un EOF (es. pipe stdin) causa loop infinito |
| Limiti hardcoded | 🟡 Medio | `limit=10` per search, `[:3]` per contatti — non configurabili |
| `_parse_revenue_mln()` silente | 🟡 Medio | Eccezione `ValueError` → `return 0.0` senza logging — revenue mancante appare come zero |

**Nota positiva:** il design complessivo è pulito e minimalista. Local import per evitare circular dependency (`from primer_ops.lead_input import LeadInput`) è gestito correttamente.

### `primer.py` — 🔴 Monolite + valori hardcoded

**Scopo:** Orchestrator principale del pipeline di generazione.

**Dimensione:** 1020 LOC in una singola funzione `generate_primer()` (noto, intenzionale per ora).

**Problemi:**

| Issue | Severità | Descrizione |
|-------|----------|-------------|
| Model names hardcoded | 🟠 Alto | `"gpt-5.2"` e `"o4-mini-deep-research"` sono fallback hardcoded in `primer.py` — dovrebbero stare in `config.py` come `get_base_model()` e `get_deep_model()` |
| `sources_payload` untyped | 🟠 Alto | `dict[str, Any]` con struttura nidificata — accesso a `step_entry["response_text"]` è fragile. Un `@dataclass` o `TypedDict` eliminerebbe KeyError silenti |
| Error handling incoerente in loop | 🟠 Alto | `except Exception as err: step_entry["error"] = ...` nel loop per-step — l'esecuzione continua silenziosamente. L'utente non sa che uno step ha fallito criticamente fino a che non legge `sources.json` |
| Path resolution duplicata | 🟡 Medio | La logica di risoluzione path assoluto (`if not path.is_absolute(): repo_root / path`) appare 3 volte: `primer.py:488`, `primer.py:379`, `primer.py:102` |
| `total_steps = 5` hardcoded | 🟡 Medio | Il contatore `[1/5]`, `[2/5]` è hardcoded — se si aggiunge una fase, il numero non si aggiorna |
| Reasoning effort mappato via string match | 🟡 Medio | `"Thinking - Reasoning: Extended"` → `"xhigh"` tramite if/elif — un dict mapping sarebbe più leggibile e testabile |
| `parse_sheet_filter()` nested | 🟡 Medio | Funzione interna a `generate_primer()` — impossibile testare in isolamento (gap noto in `CLAUDE.md`) |
| `### CONTEXT` injection fragile | 🟡 Medio | Se il prompt non contiene `### CONTEXT`, il contesto viene prepended blindly — possibile duplicazione se la struttura del prompt è inconsistente tra fogli |

---

### `render_docx.py` — ⚠️ Complesso ma funzionale

**Scopo:** Conversione Markdown → DOCX tramite `markdown-it-py` con fallback parser custom.

**Dimensione:** 1317 LOC con ~100+ funzioni helper.

**Problemi:**

| Issue | Severità | Descrizione |
|-------|----------|-------------|
| `paragraph.text = updated` | 🟡 Medio | Assignment diretto a `.text` cancella tutti i `Run` con il loro formatting. L'aggiornamento di placeholder nel template DOCX (`{{COMPANY}}`, `{{DATE}}`) perde grassetto/corsivo/font del template originale |
| `_is_markdown_it()` fragile | 🟡 Medio | `parser.__class__.__name__ == "MarkdownIt"` — se la classe viene rinominata o subclassata, il check fallisce. Meglio `isinstance(parser, MarkdownIt)` |
| Silent `except KeyError: pass` | 🟡 Medio | Stile non trovato nel documento → silenzio. Dovrebbe essere almeno `logger.warning()` |
| Silent `except Exception: return False` | 🟡 Medio | Creazione hyperlink fallita → silenzio. Stessa issue |
| Image rendering non implementato | 🟡 Medio | `"TODO: image omitted"` come placeholder — `pillow` è in dipendenze ma non usato per questo scopo |
| `_compute_fence_mask` e `_compute_table_mask` | 🟢 Basso | Logica quasi identica, potrebbe consolidarsi |
| `"Name"` / `"Description"` heuristic | 🟢 Basso | Layout tabella 30/70 hardcoded solo se header è esattamente `name` / `description` — funziona per il template corrente ma fragile |

**Nota positiva:** la normalizzazione markdown (`normalize_markdown_for_docx`) è robusta — gestisce correttamente fence blocks, separatori tabella, heading numerici con heuristic sensata.

Wrapper minimale. Carica `.env`, setta default per `PRIMER_WORD_TEMPLATE_PATH`, importa dinamicamente il CLI. Nessun problema.

## 10. Conclusioni

Il progetto è **ben concepito e correttamente strutturato** per la fase attuale. L'architettura modulare, il design delle scritture atomiche, e l'integrazione HubSpot mostrano buona ingegneria. La documentazione interna (`CLAUDE.md`, `README.md`, `docs/review.md`) è eccellente per un progetto in early stage.

**Il rischio più immediato** è il silent I/O failure in `io_helpers.py`: è un bug di 5 minuti da risolvere che in produzione può causare la perdita silenziosa dell'output senza alcun indicatore per l'utente.

**Il debito più significativo** per la Fase 2 è la mancanza di typing su `sources_payload` — prima di aggiungere pitch, LOP e pricing, vale la pena tipizzare questa struttura dati centrale per evitare di replicare la stessa fragilità nelle nuove features.

**Coverage al 55%** è accettabile per uso interno supervisionato. Prima del go-live su volumi più alti, le aree a 0% (retry, HubSpot, filtering) dovrebbero essere coperte.
